# ViralScope AI PRD

## 1. 项目目标

ViralScope AI 是一个面向跨境电商和出海品牌运营场景的两天 Demo 项目。

它不是完整产品，也不是产品经理式平台方案，而是一个可运行的本地 AI workflow：

> 人工筛选值得拆解的热点视频 -> 把视频放入指定文件夹 -> 手动填写 JSON 配置 -> 运行脚本自动抽帧 -> AI 生成爆款拆解报告 -> 沉淀可复用内容公式和翻拍建议。

项目核心目标：

1. 展示数据分析与策略优化能力
2. 展示对达人内容、爆款结构、转化问题的理解
3. 展示用 AI 提升运营复盘效率的能力
4. 在两天内完成一个能演示、能讲清楚、能放进作品集的 MVP

## 2. 面向岗位

该项目主要服务于以下出海运营相关岗位的作品集展示：

- 跨境电商运营
- TikTok Shop 运营
- KOL / 达人运营
- 商家运营
- 行业运营
- 内容策略运营
- 出海品牌 Campaign 运营

项目表达优先级：

1. 数据分析和策略优化
2. 产品用户 sense
3. 业务理解
4. AI 提效

不把项目包装成产品经理作品，避免和产品岗候选人正面竞争。

## 3. 核心业务痛点

### 痛点 1：人工拆解热点视频效率低

运营人员需要大量查看热点视频、截图、总结 Hook、分析结构、提炼可复用打法。这个过程耗时，且不同人的判断标准不统一。

### 痛点 2：爆款经验难以沉淀

团队经常知道“某条视频火了”，但没有把它转化成可复用的达人 brief、脚本结构和 Campaign 内容策略。

### 痛点 3：高播放不等于高转化

部分达人视频播放和互动不错，但不出单。问题可能来自受众不匹配、产品露出太晚、利益点不清晰、CTA 弱、信任背书不足等。

### 痛点 4：运营复盘缺少结构化输出

热点内容、达人投稿、转化结果和复盘结论容易散落在聊天记录、文件夹、Excel 或飞书文档中，难以形成长期可复用的内容资产。

## 4. 两天 Demo 边界

### 4.1 Demo 必须完成

两天内只完成能支撑作品集和面试演示的核心闭环：

1. 指定文件夹读取本地视频
2. 读取每个视频对应的 JSON 配置文件
3. 自动抽帧
4. 生成截图时间轴
5. AI 生成结构化爆款拆解报告
6. Markdown 报告输出
7. 转化诊断框架展示
8. 一个手动验证案例

### 4.2 Demo 明确不做

以下功能不进入两天 Demo 的必须范围：

| 不做事项 | 原因 |
|---|---|
| 网页上传界面 | 与核心卖点无关，容易消耗前端调试时间 |
| 网页表单 / GUI 表单 | 改用 JSON 配置，零 UI 成本且更稳定 |
| ASR 自动口播转写 | 工程复杂度高，容易被音质、口音、背景音乐影响；对作品集核心说服力有限 |
| 多 provider API 配置系统 | 过度设计；Demo 只需要一个 OpenAI-compatible 配置即可 |
| Excel 多 Sheet 导出 | 调试成本高，边际展示价值低；Markdown 更适合快速展示和修改 |
| TikTok 链接解析 | 涉及平台限制和稳定性问题，不适合两天 Demo |
| 自动抓取点赞/评论/收藏 | 平台数据口径和权限不稳定，MVP 阶段手动填写更真实 |
| OCR 自动识别字幕 | 可作为 Roadmap，不影响核心拆解流程 |
| 批量视频处理产品化 | 单视频流程先跑通，批量化放到后续版本 |
| 完整转化归因模型 | 两天内只做诊断框架，不做因果判断 |

## 5. 核心用户流程

### Step 1：运营人工筛选视频

运营人员先人工判断哪些热点视频值得拆解，然后将视频下载到本地。

该步骤保留人工判断，因为运营场景中“是否值得拆解”本身需要业务判断，不应完全交给 AI。

### Step 2：把视频放入指定文件夹

用户把视频文件放入项目目录，例如：

```text
samples/
  video1.mp4
  video1.json
```

支持格式：

- `.mp4`
- `.mov`
- `.m4v`
- `.webm`

### Step 3：填写 JSON 配置文件

每个视频对应一个 JSON 文件，用来补充运营背景、达人画像和数据表现。

示例：

```json
{
  "video": "video1.mp4",
  "platform": "TikTok",
  "category": "3C",
  "product_type": "Webcam",
  "target_audience": "居家办公人群、主播、线上会议用户",
  "campaign_type": "新品冷启动",
  "title": "Why your video calls look unprofessional",
  "speech_summary": "开头吐槽电脑摄像头像素差，随后展示外接摄像头前后对比效果。",
  "creator_profile": {
    "country": "美国",
    "language": "英语",
    "niche": "科技测评 / 办公效率",
    "follower_tier": "中腰部达人",
    "style": "专业测评、口语化、强对比",
    "audience_fit": "高"
  },
  "metrics": {
    "views": 120000,
    "likes": 8500,
    "comments": 430,
    "saves": 1200,
    "shares": 300,
    "completion_rate": "28%",
    "orders": 12,
    "gmv": "960 USD",
    "sample_cost": "60 USD"
  },
  "operator_note": "播放高但成交一般，评论区多问价格和兼容性。"
}
```

### Step 4：运行脚本处理

用户通过命令行运行脚本，例如：

```powershell
python analyze_video.py samples/video1.json
```

脚本读取视频和 JSON 配置，完成抽帧、AI 分析和 Markdown 输出。

### Step 5：系统自动抽帧

系统按照短视频内容分析逻辑抽帧：

- 0-3 秒：每 0.5 秒截取 1 帧
- 3-15 秒：每 1 秒截取 1 帧
- 15 秒后：每 2 秒截取 1 帧

保留该设计，因为短视频的前 3 秒决定停留，后续重点看内容结构、产品露出和 CTA。

### Step 6：AI 生成爆款拆解报告

系统把截图时间轴、JSON 配置中的运营背景、达人画像、数据表现和口播摘要一起提交给 AI，生成结构化拆解报告。

### Step 7：输出 Markdown 报告

系统输出 `.md` 报告，例如：

```text
outputs/
  video1/
    frames/
    ViralScope_report.md
```

Markdown 是 Demo 阶段的首选输出格式，因为：

- 生成稳定
- 方便修改
- 适合作品集展示
- 可复制到飞书、Notion、GitHub 或网页

### Step 8：展示转化诊断延伸模块

转化诊断不作为完整产品实现，只做轻量框架：

- 输入播放量、互动表现、订单 / GMV 等手动数据
- 输出可能的低转化原因
- 给出达人可执行反馈建议

该模块用于展示业务延伸思路，而不是宣称已经完成准确归因。

## 7. 抽帧规则

### 7.1 抽帧策略

| 视频区间 | 抽帧频率 | 分析目的 |
|---|---|---|
| 0-3 秒 | 每 0.5 秒 1 帧 | 分析首屏、Hook、字幕、动作变化 |
| 3-15 秒 | 每 1 秒 1 帧 | 分析正文展开、产品露出、卖点证明、节奏变化 |
| 15 秒后 | 每 2 秒 1 帧 | 分析后段结构、补充信息、CTA、评论引导 |

### 7.2 截图时间轴输出

脚本在输出目录保存截图，并在 Markdown 报告中列出时间轴。

示例：

| 时间点 | 截图文件 | 运营观察 |
|---|---|---|
| 0.0s | frame_000.jpg | 首屏是否有强视觉刺激 |
| 0.5s | frame_001.jpg | 是否出现痛点、反差或利益点 |
| 1.0s | frame_002.jpg | 字幕是否足够明确 |
| 2.5s | frame_005.jpg | 黄金 3 秒是否完成悬念建立 |
| 5.0s | frame_006.jpg | 是否进入主体内容 |
| 7.0s | frame_007.jpg | 产品是否清晰露出 |

## 8. AI 分析模板

### 8.1 系统角色

```text
你是一个跨境电商达人营销内容策略分析师。
你的任务是帮助运营人员拆解短视频为什么有流量，以及它是否有转化潜力。

请基于：
1. 视频截图时间轴
2. 用户填写的视频基础信息
3. 达人画像
4. 数据表现
5. 口播 / 字幕摘要

输出结构化分析。

要求：
- 语言直白
- 不要空谈理论
- 每个结论都要对应可执行动作
- 必须区分“流量原因”和“转化原因”
- 必须给出达人可直接理解的反馈建议
- 如果信息不足，要明确说明需要补充什么，而不是编造
```

### 8.2 报告输出结构

AI 报告使用英文正文和英文标题，固定为以下 9 个模块（括号内中文为说明，非实际标题）：

1. `## 1. Basic Info`（基础信息总结）
2. `## 2. Why It Went Viral`（爆款核心原因，区分流量原因/转化原因，3-5 条可复制点）
3. `## 3. First 3-Second Hook`（开头 3 秒 Hook 拆解）
4. `## 4. Content Structure`（内容结构拆解）
5. `## 5. Emotional Value`（情绪价值拆解）
6. `## 6. Repeatable Viral Formula`（可直接照搬的爆款公式）
7. `## 7. Remake & Optimization`（翻拍优化建议），含三个子块：
   - `### Copy Directly (3)`（可以直接照搬的 3 个点）
   - `### Do Better (2)`（可以做得更好的 2 个点）
   - `### Our Brand Version`（适合我们账号的落地版本）
8. `## 8. Conversion Diagnosis`（转化诊断初步判断）
9. `## 9. Creator Feedback`（达人反馈话术）

转化诊断由 `diagnose.py` 单独生成 `diagnosis.md`，为英文正文，固定 4 节：
`## 1. Type Judgment` / `## 2. One-Sentence Conclusion` / `## 3. Top Priority Actions` / `## 4. Creator Feedback`。

### 8.3 “像样报告”的验收标准

为避免反复调 Prompt 但不知道什么时候算完成，报告质量用以下标准验收：

1. 每条爆款原因必须引用具体时间点或画面细节作为依据，不能只写空泛判断。
2. 翻拍建议必须是可以直接发给达人执行的一句话指令，不能写“建议加强产品露出”这类正确但无用的话。
3. 必须区分流量原因和转化原因，不能把“播放高”直接等同于“能出单”。
4. 至少输出 3 条可复制点、2 条可优化点、1 个适合品牌账号的落地版本。
5. 如果信息不足，必须说明缺少什么信息，不能假装已经准确判断。

错误示例：

> 建议优化内容节奏，提升用户转化。

正确示例：

> 建议把产品前后画质对比提前到前 3 秒内出现，不要等到第 8 秒。当前视频先铺垫太久，用户可能已经划走；如果开头直接展示“电脑原生摄像头 vs 外接摄像头”的对比，更容易让目标用户理解购买理由。

## 10. 转化诊断轻量模块

### 10.1 模块定位

转化诊断模块是工具 1 的延伸，不是两天 Demo 的核心。

它用于回答：

> 为什么一条视频播放和互动不错，但没有带来订单？

### 10.2 输入

| 字段 | 示例 |
|---|---|
| 播放量 | 120000 |
| 点赞率 | 7.1% |
| 评论率 | 0.36% |
| 收藏率 | 1.0% |
| 订单量 | 12 |
| 商品单价 | 79 USD |
| 达人垂类 | 泛娱乐 / 科技测评 / 生活方式 |
| 产品露出时间 | 第 8 秒 |
| CTA 是否明确 | 否 |
| 评论区主要问题 | 价格、兼容性、购买链接 |

### 10.3 输出

| 诊断项 | 可能问题 | 优化动作 |
|---|---|---|
| 受众匹配 | 泛流量较多，购买意图弱 | 优先找科技测评 / 办公效率类达人 |
| 产品露出 | 产品出现太晚 | 前 3 秒展示使用前后对比 |
| 利益点 | 只展示好看，没有说明购买理由 | 改成场景化利益，如开会更专业、直播画质更清晰 |
| CTA | 没有明确引导 | 结尾加入评论关键词、限时折扣或购买路径说明 |
| 评论区承接 | 用户疑问未被回答 | 在置顶评论回答价格、兼容性和安装方式 |

### 10.4 Demo 展示方式

两天 Demo 中只做一个 mock case：

> 某达人视频播放高、互动高，但订单低。系统根据用户填写的数据和内容拆解结果，输出可能原因和达人反馈话术。

不宣称系统能准确归因，只说它可以帮助运营快速提出诊断假设。

## 11. 两天执行计划

### Day 1：跑通核心流程

目标：指定文件夹读取视频、读取 JSON、抽帧、生成时间轴。

任务：

1. 搭建项目目录
2. 准备 `samples/` 和 `outputs/` 文件夹
3. 编写 `video1.json` 样例配置
4. 实现读取 JSON
5. 实现本地视频抽帧
6. 保存截图和时间轴信息
7. 准备 1 条测试视频和 1 组备用截图

Day 1 成功标准：

- 运行脚本后能生成截图文件夹
- 能生成包含时间轴的基础 Markdown
- 不依赖 AI，也能展示“视频 -> 截图 -> 结构化输入”的基础流程

### Day 2：打磨 AI 报告和作品集叙事

目标：生成一份有业务判断的爆款拆解报告。

任务：

1. 接入一个 OpenAI-compatible API
2. 编写并调试分析 Prompt
3. 生成 Markdown 报告
4. 增加转化诊断 mock case
5. 准备演示样例
6. 准备面试讲法

Day 2 成功标准：

- 能生成一份结构完整、建议具体的 Markdown 报告
- 报告满足第 8.3 节质量验收标准
- 能解释为什么转化诊断先做框架，而不是完整自动化

## 12. 风险清单与降级方案

| 风险 | 判断标准 | 降级方案 |
|---|---|---|
| 视频抽帧失败 | 1 小时内无法稳定生成截图 | 使用提前准备好的截图文件夹，保留时间轴展示 |
| API 调不通 | 30 分钟内无法完成调用 | 脚本生成 Prompt，复制到 ChatGPT / Claude 手动生成报告 |
| AI 输出太空 | 报告只讲理论，没有具体动作 | 在 Prompt 中加入错误示例和正确示例，强制输出可执行动作 |
| Markdown 导出失败 | 30 分钟内无法保存文件 | 控制台直接打印报告，手动复制作为演示 |
| 命令行脚本体验差 | 新手操作容易出错 | 写固定命令和样例文件，不做交互式输入 |
| 转化诊断做不完 | 影响主流程进度 | 只保留 mock case 和 Roadmap 文案 |
| 测试视频效果不好 | AI 无法看出明显结构 | 准备第二条更典型的视频或使用预设截图 |
| 时间不够 | Day 2 上午仍未跑通主流程 | 砍掉 API 自动调用，保留 Prompt 生成和手动 AI 输出 |

## 13. Roadmap

### V1：两天 Demo

- 指定文件夹读取视频
- JSON 结构化输入
- 分段抽帧
- AI 爆款拆解
- Markdown 报告
- 转化诊断 mock case

### V2：运营样本库

- 文件夹批量导入
- 多视频横向对比
- 按平台、品类、达人画像筛选
- 爆款公式沉淀

### V3：自动化增强

- ASR 自动口播转写
- OCR 识别字幕
- Excel 导出
- BGM / 音效分析
- 支持更多模型 provider
- 网页界面或轻量 GUI

### V4：转化诊断增强

- 接入订单 / GMV 数据
- 识别高播放低转化视频
- 输出达人二次修改 brief
- 支持 Campaign 复盘看板

## 14. 面试表达版本

可以这样讲：

> 我做这个项目不是为了展示一个完整产品，而是想把海外达人运营里一个非常高频的工作流做成 AI Demo：人工挑出值得拆解的热点视频后，把视频放进指定文件夹，我用 JSON 补充达人画像、视频数据和运营背景，脚本自动抽帧，AI 根据统一模板生成爆款拆解报告。
>
> 这个项目的重点不是技术堆叠，而是把“为什么这条视频有流量”“哪些点可以复用”“如果高播放但低转化可能卡在哪里”这几个运营问题结构化。两天 Demo 里我主动砍掉了网页上传、ASR、多模型配置、Excel 多 Sheet 这些工程成本高但不直接影响展示效果的功能，优先保证爆款拆解质量。
>
> 后续如果继续做，我会把它扩展成达人内容样本库，再接订单和 GMV 数据，用于识别高播放低转化的视频，并生成达人二次优化建议。

## 15. 项目成功标准

这个 Demo 成功与否，不看功能数量，而看以下 5 点：

1. 是否能稳定跑通“视频 -> 抽帧 -> AI 分析 -> 报告”的闭环
2. 报告是否体现真实达人运营判断
3. 建议是否具体到达人能直接修改内容
4. 是否能解释为什么先做爆款拆解，再延伸到转化诊断
5. 是否能在面试中清楚体现数据分析、策略优化和业务理解
